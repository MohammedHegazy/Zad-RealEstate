# الفصــل السادس

# التحقيق والتنفيذ

## 6.1. مقدمة

يوضح هذا الفصل كيفية تنفيذ مشروع زاد للعقارات عملياً بعد الانتهاء من التحليل والتصميم. تم تقسيم المشروع إلى واجهة أمامية داخل مجلد `project-RealEstate`، وواجهة خلفية داخل مجلد `project-RealEstate_database`، إضافة إلى خدمة تعلم آلي داخل `ml/pricing` للتنبؤ بسعر العقار.

اعتمد التنفيذ على فصل واضح بين الواجهة والخادم. فالواجهة الأمامية تعرض الصفحات وتتفاعل مع المستخدم، بينما يقدم Laravel واجهات API وينفذ منطق الأعمال ويتعامل مع قاعدة البيانات. أما خدمة Flask فتستقبل بيانات العقار وتعيد السعر المتوقع عند الحاجة.

## 6.2. بيئة وتقنيات التنفيذ

تم تنفيذ الواجهة الأمامية باستخدام Vue وVite، مع الاعتماد على Pinia لإدارة الحالة وVue Router للتنقل بين الصفحات. كما تم استخدام Bootstrap في تنسيق الواجهات، وLeaflet مع OpenStreetMap لعرض الخرائط والمواقع العقارية.

أما الواجهة الخلفية فتم تنفيذها باستخدام Laravel 12، مع Laravel Sanctum لحماية المسارات الخاصة بالمستخدمين. وتم تنظيم الكود إلى مسارات، متحكمات، نماذج، طلبات تحقق، خدمات، واختبارات. وتستخدم خدمة التنبؤ السعري Flask وscikit-learn وملفات نموذج محفوظة بصيغة `pkl`.

| الجزء | التقنية | الدور |
|---|---|---|
| الواجهة الأمامية | Vue + Vite | عرض الصفحات والتفاعل مع المستخدم |
| إدارة الحالة | Pinia | حفظ حالة المستخدم والبيانات المشتركة |
| التنقل | Vue Router | إدارة المسارات العامة والمحمية والإدارية |
| الواجهة الخلفية | Laravel | API ومنطق الأعمال وقاعدة البيانات |
| المصادقة | Laravel Sanctum | حماية المسارات الخاصة |
| الخرائط | Leaflet + OpenStreetMap | عرض العقارات جغرافياً |
| التنبؤ السعري | Flask + scikit-learn | تقدير سعر العقار |

## 6.3. مسار العمل العام في النظام

يبدأ مسار العمل غالباً من الواجهة الأمامية، حيث يفتح المستخدم صفحة أو يضغط على زر معين. ترسل Vue طلباً إلى Laravel API، ثم يتحقق Laravel من البيانات والصلاحيات، وينفذ المتحكم أو الخدمة المناسبة، ويتعامل مع قاعدة البيانات عند الحاجة، ثم يعيد استجابة JSON إلى الواجهة.

في العمليات العادية، مثل عرض العقارات أو حفظ المفضلة، تبقى العملية بين Vue وLaravel وقاعدة البيانات. أما في عملية التنبؤ السعري، فإن Laravel يرسل البيانات إلى خدمة Flask، وتعيد الخدمة السعر المتوقع ليتم عرضه للمستخدم أو حفظه في جدول التنبؤات.

## 6.4. تنفيذ المصادقة والأدوار

تم تنفيذ المصادقة باستخدام Laravel Sanctum. يستطيع المستخدم إنشاء حساب، تسجيل الدخول، الحصول على رمز وصول، ثم استخدامه لتنفيذ العمليات المحمية مثل المفضلة، الرسائل، التفضيلات، المحافظ، التحليلات، والتقييمات.

تم فصل المسارات إلى مسارات عامة يستطيع الزائر الوصول إليها، ومسارات محمية تحتاج إلى تسجيل دخول، ومسارات إدارية تحتاج إلى صلاحية مدير. يساعد هذا الفصل على حماية البيانات الحساسة ومنع تنفيذ عمليات إدارية من مستخدم عادي.

تشمل عمليات المصادقة الأساسية:

- تسجيل مستخدم جديد مع التحقق من البيانات.
- تسجيل الدخول وإرجاع بيانات المستخدم ورمز الوصول.
- تسجيل الخروج وإلغاء الرمز المستخدم.
- حماية المسارات الخاصة باستخدام `auth:sanctum`.
- حماية مسارات الإدارة باستخدام وسيط خاص بالمدير.

## 6.5. تنفيذ إدارة الأدوار ولوحات التحكم

يدعم النظام أكثر من نوع مستخدم، مثل المستخدم العادي، المستثمر، المالك، الشركة، الوسيط، والمدير. لا تظهر هذه الأدوار كأسماء فقط، بل تنعكس على طريقة الوصول إلى الواجهات والمسارات. فالمدير يمتلك لوحة تحكم خاصة، بينما يستخدم المستخدم العادي الصفحات العامة والمحمية.

تتضمن لوحة الإدارة صفحات لإدارة المستخدمين، العقارات، الشركات، الوسطاء، المدن، المناطق، التقييمات، طلبات التوثيق، والإحصاءات. ويهدف هذا الجزء إلى تمكين المدير من مراجعة المحتوى والتحكم في حالة العناصر المهمة داخل النظام.

## 6.6. تنفيذ إدارة العقارات والوسائط

تم تنفيذ العقارات ككيان رئيسي داخل النظام. يحتوي العقار على بيانات مثل النوع، العنوان، السعر، الموقع، المدينة، المنطقة، الحالة، الوصف، والارتباط بالمالك أو الجهة العقارية. وتوجد مسارات عامة لعرض العقارات ومسارات محمية أو إدارية لإدارتها.

يدعم النظام وسائط العقار من خلال صور وفيديوهات وإعلانات مرتبطة به. يتم رفع الملفات ومعالجتها عبر خدمة مخصصة، مما يجعل إدارة الملفات منفصلة عن منطق العقارات نفسه. هذا الأسلوب يسهل لاحقاً إضافة ضغط للصور أو تخزين خارجي أو قيود إضافية على الملفات.

كما يدعم النظام حالات مختلفة للعقار، مثل القبول أو الرفض أو المراجعة، بحيث يستطيع المدير التحكم في ظهور العقارات للمستخدمين.

## 6.7. تنفيذ المدن والمناطق والبحث الجغرافي

تم تنفيذ المدن والمناطق كجزء أساسي من بنية العقارات. فكل عقار يرتبط بموقع جغرافي يساعد المستخدم على البحث والتصفية. كما تعرض الواجهة العقارات ضمن صفحات المدن والمناطق، وتستخدم الخرائط لتوضيح الموقع.

يدعم النظام البحث الجغرافي من خلال خدمات خلفية تتعامل مع الإحداثيات والمسافات. وهذا يسمح بعرض العقارات القريبة أو المطابقة لموقع معين، وهو أمر مهم في المنصات العقارية لأن الموقع عامل رئيسي في القرار.

## 6.8. تنفيذ الشركات والوسطاء

تم تنفيذ الشركات العقارية والوسطاء ككيانات مستقلة مرتبطة بالمستخدمين والعقارات. تستطيع الشركة امتلاك ملف خاص يحتوي على بياناتها وروابط التواصل الخاصة بها، كما يمكن ربط الوسطاء بها.

يدعم النظام عرض الشركات والوسطاء للمستخدمين، كما يدعم إدارتهم من لوحة المدير. هذا يضيف طبقة تنظيمية مهمة للمنصة، لأن العقارات لا تأتي دائماً من أفراد، بل قد تأتي من شركات ووسطاء يحتاجون إلى إدارة وتوثيق.

## 6.9. تنفيذ المفضلة والتوصيات

توفر المفضلة للمستخدم إمكانية حفظ العقارات أو الوسطاء للرجوع إليها لاحقاً. يتم حفظ العلاقة بين المستخدم والعنصر المختار في قاعدة البيانات، ثم تعرض الواجهة قائمة المفضلة في صفحة خاصة.

أما التوصيات فتعتمد على تفضيلات المستخدم وتفاعلاته مع العقارات. يحفظ النظام اهتمامات المستخدم مثل النوع أو السعر أو الموقع، كما يسجل بعض التفاعلات، ثم يستخدم هذه البيانات لتوليد عقارات مقترحة. تم فصل منطق التوصيات في خدمات خاصة، مما يجعل تطويره لاحقاً أسهل.

## 6.10. تنفيذ الاستثمار والمحافظ

يدعم المشروع الجانب الاستثماري من خلال خدمات لحساب المؤشرات المالية، مثل العائد المتوقع وفترة الاسترداد وبعض بيانات الجدوى. كما يستطيع المستخدم إنشاء تحليل استثماري لعقار معين وحفظ النتيجة.

تم تنفيذ المحافظ العقارية لتجميع عدة عقارات ضمن محفظة واحدة. يستطيع المستخدم إنشاء محفظة، إضافة عقارات إليها، ومتابعة بياناتها. كما توجد لوحة للمستثمر تعرض ملخصاً يساعد على فهم الوضع الاستثماري بدلاً من عرض كل عقار بشكل منفصل.

## 6.11. تنفيذ الرسائل والإشعارات

يدعم النظام الرسائل بين المستخدمين أو الجهات المرتبطة بالعقارات. عند إرسال رسالة، يحفظ النظام محتواها ويربطها بالأطراف المناسبة، ثم يمكن للطرف الآخر فتح المحادثة ومتابعتها.

كما يدعم النظام الإشعارات لإعلام المستخدم بوجود حدث مهم، مثل رسالة جديدة أو تحديث مرتبط بحسابه أو نشاطه. وتستطيع الإدارة مراقبة بعض الرسائل أو إدارتها حسب المسارات الإدارية المتاحة.

## 6.12. تنفيذ الثقة والمراجعات والتوثيق

تم تنفيذ مراجعات للعقارات والوسطاء والشركات، بحيث يستطيع المستخدم تقييم الجهة أو العقار وكتابة رأيه. تساعد هذه المراجعات على رفع مستوى الشفافية داخل المنصة.

كما يدعم النظام طلبات التوثيق، حيث يمكن لجهة معينة تقديم طلب، ثم يراجعه المدير ويغير حالته. وترتبط هذه العملية بدرجة الثقة، وهي عنصر مهم في المجال العقاري لأن المستخدم يحتاج إلى مؤشرات تساعده على الوثوق بالجهات التي يتعامل معها.

## 6.13. تنفيذ التنبؤ السعري وخدمة ML

تم تنفيذ التنبؤ السعري من خلال خدمة مستقلة مبنية باستخدام Flask. يرسل Laravel بيانات العقار إلى هذه الخدمة، فتقوم بتجهيز البيانات وتمريرها إلى نموذج scikit-learn، ثم تعيد السعر المتوقع.

يوفر Laravel مسارات وخدمات للتعامل مع هذا الجزء، مثل إرسال بيانات عقار موجود أو إجراء توقع مسبق. كما يمكن حفظ نتائج التنبؤ في جدول `price_predictions` عند الحاجة، مما يسمح بالرجوع إليها أو تحليلها لاحقاً.

فصل خدمة ML عن Laravel يجعل تطوير نموذج التنبؤ أسهل، لأن تحسين النموذج أو تدريبه ببيانات جديدة لا يتطلب تغيير بنية الواجهة الخلفية بالكامل.

## 6.14. تنفيذ الواجهة الأمامية

تم تنظيم الواجهة الأمامية إلى صفحات عامة، صفحات محمية، وصفحات إدارية. الصفحات العامة تشمل العقارات، تفاصيل العقار، المدن، المناطق، الشركات، والوسطاء. الصفحات المحمية تشمل الملف الشخصي، المفضلة، التوصيات، الرسائل، الاستثمار، والمحافظ. أما صفحات الإدارة فتغطي إدارة المستخدمين والعقارات والشركات والوسطاء والثقة والمحتوى.

تستخدم الواجهة ملفات API للتعامل مع Laravel، كما تستخدم مكونات مشتركة لتقليل التكرار وتحسين التنظيم. ويعتمد التنقل على Vue Router، بينما تساعد Pinia على إدارة حالة المستخدم والبيانات المشتركة.

## 6.15. الخدمات البرمجية الأساسية

اعتمد التنفيذ الخلفي على مجموعة خدمات متخصصة بدلاً من وضع كل المنطق داخل المتحكمات. من هذه الخدمات:

- خدمة رفع الملفات.
- خدمة البحث الجغرافي.
- خدمات التوصيات.
- خدمة حساب الاستثمار.
- خدمة لوحة المستثمر.
- خدمة المحافظ.
- خدمة الثقة.
- خدمة التنبؤ السعري.
- خدمة اتجاهات السوق.

هذا التنظيم يجعل المشروع أسهل في الفهم، لأن كل خدمة مسؤولة عن مجال محدد داخل النظام.

## 6.16. مقارنة مختصرة بين مسارات التنفيذ

| المسار | طريقة التنفيذ |
|---|---|
| الزائر | يستخدم الصفحات العامة لعرض العقارات والمدن والشركات والوسطاء. |
| المستخدم المسجل | يستخدم المسارات المحمية للمفضلة والرسائل والتوصيات والتقييمات. |
| المالك أو الشركة | يدير العقارات أو الملف الخاص بالجهة حسب الصلاحيات المتاحة. |
| المدير | يستخدم لوحة الإدارة لمراجعة المستخدمين والمحتوى والحالات والتوثيق. |
| التنبؤ السعري | ينتقل الطلب من Vue إلى Laravel ثم إلى Flask وتعود النتيجة للواجهة. |

## 6.17. خلاصة الفصل

أظهر تنفيذ المشروع أن النظام مبني بشكل منظم حسب المجالات. فقد تم تنفيذ المصادقة، العقارات، الوسائط، المواقع، الشركات، الوسطاء، المفضلة، التوصيات، الاستثمار، المحافظ، الرسائل، الإشعارات، الثقة، التقييمات، التوثيق، التنبؤ السعري، والواجهة الأمامية.

كما أن فصل الواجهة عن الخادم، وفصل خدمة التنبؤ السعري عن Laravel، واستخدام خدمات خلفية متخصصة، كلها قرارات تجعل المشروع قابلاً للتوسع والصيانة في المراحل القادمة.
