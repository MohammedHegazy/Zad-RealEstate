# الفصــل الرابع

# الدراسة التصميمية

## 4.1. مقدمة

توضح الدراسة التصميمية الطريقة التي تم بها تحويل نتائج التحليل إلى بنية قابلة للتنفيذ. فبعد تحديد المتطلبات وحالات الاستخدام في الفصل السابق، يبين هذا الفصل تصميم النظام من حيث مخططات التسلسل، قاعدة البيانات، البنية المعمارية، وتصميم الواجهات.

يعتمد مشروع زاد للعقارات على تصميم متعدد الطبقات. الواجهة الأمامية مبنية باستخدام Vue وVite لعرض الصفحات والتعامل مع المستخدم، والواجهة الخلفية مبنية باستخدام Laravel لتوفير API ومنطق الأعمال والمصادقة، بينما تم فصل خدمة التنبؤ السعري في خدمة Flask مستقلة تعتمد على نموذج تعلم آلي.

يساعد هذا الفصل بين الطبقات على جعل النظام أوضح وأسهل في الصيانة. فالواجهة لا تتعامل مباشرة مع قاعدة البيانات، بل ترسل الطلبات إلى Laravel. والخادم الخلفي لا يدمج منطق التنبؤ السعري داخله بالكامل، بل يتواصل مع خدمة خارجية عند الحاجة.

## 4.2. مخططات التسلسل

توضح مخططات التسلسل طريقة تفاعل المستخدم والواجهة الأمامية والواجهة الخلفية والخدمات الداخلية أثناء تنفيذ العمليات المهمة. تبدأ معظم العمليات من المستخدم داخل صفحة Vue، ثم يرسل المتصفح طلباً إلى API، وبعدها ينفذ Laravel التحقق والمنطق المطلوب، ثم يعيد النتيجة إلى الواجهة.

أهم التسلسلات في المشروع هي:

| التسلسل | الوصف المختصر |
|---|---|
| جلب قائمة التوصيات | يطلب المستخدم التوصيات، فيقرأ النظام التفضيلات والتفاعلات ثم يعيد عقارات مناسبة. |
| تقييم عقار داخل محرك التوصيات | يقارن النظام خصائص العقار مع تفضيلات المستخدم لحساب درجة الملاءمة. |
| إعادة توليد التوصيات | يعيد النظام بناء الاقتراحات عند تغير تفضيلات المستخدم أو تفاعلاته. |
| العقارات المشابهة | يعرض النظام عقارات قريبة في النوع أو الموقع أو السعر من العقار الحالي. |
| إنشاء عقار مع حساب ROI | يدخل المالك بيانات العقار، ثم يحفظ النظام العقار ويحسب مؤشرات مالية عند توفر المعطيات. |
| حفظ تحليل استثماري | يرسل المستخدم بيانات التحليل، فينفذ الخادم الحسابات ويحفظ النتيجة. |
| لوحة المستثمر | تجمع الواجهة بيانات المحافظ والتحليلات والعقارات الاستثمارية وتعرضها للمستخدم. |
| إضافة عقار إلى محفظة | يختار المستخدم محفظة وعقاراً، ثم يحفظ النظام العلاقة بينهما. |
| إرسال رسالة وإشعار | تحفظ الرسالة في قاعدة البيانات ويرسل النظام إشعاراً للطرف المستلم. |
| فتح محادثة | يجلب النظام الرسائل السابقة بين الأطراف ويعرضها في الواجهة. |
| مراقبة الرسائل من المدير | يتيح النظام للمدير مراجعة الرسائل أو متابعة النشاط عند الحاجة. |
| تنبؤ سعر عقار موجود | يرسل Laravel بيانات العقار إلى خدمة Flask ثم يعيد السعر المتوقع. |
| التنبؤ المسبق Preview | يسمح النظام بتقدير السعر قبل حفظ العقار أو قبل اتخاذ قرار نهائي. |
| معالجة Flask داخلياً | تستقبل خدمة Flask البيانات، تجهزها للنموذج، ثم تعيد نتيجة التنبؤ. |

توضح هذه التسلسلات أن النظام لا يعتمد على عملية واحدة فقط، بل يحتوي على مسارات مختلفة: مسارات بيانات عادية، مسارات استثمارية، مسارات تواصل، ومسارات تعتمد على خدمة تعلم آلي.

## 4.3. مخطط كيانات قاعدة البيانات "ERD"

يعتمد تصميم قاعدة البيانات على ملف `ERD2.dbml`، وهو يمثل الكيانات الأساسية والعلاقات بينها. تم تقسيم الجداول حسب المجال لتسهيل فهمها وصيانتها.

### المستخدمون والتحكم بالوصول

تتضمن هذه المجموعة جداول المستخدمين والمصادقة والجلسات والرموز. الجدول الأساسي هو `users`، وترتبط به عمليات مثل تسجيل الدخول، الملف الشخصي، التفضيلات، المفضلة، الرسائل، المحافظ، والتقييمات. كما توجد جداول مساندة مثل `personal_access_tokens` و`password_reset_tokens`.

### المواقع الجغرافية

تشمل الجداول الخاصة بالمدن والمناطق، مثل `cities` و`places`. ترتبط هذه الجداول بالعقارات لتحديد موقع كل عقار ضمن مدينة ومنطقة، كما تساعد في التصفية والبحث وعرض العقارات على الخريطة.

### الشركات والوسطاء

تدعم الجداول الخاصة بـ `companies` و`agents` تمثيل الجهات العقارية داخل النظام. يمكن ربط الشركات بالوسطاء، وربط الوسطاء أو الشركات بالعقارات حسب طبيعة العمل. كما يمكن إضافة روابط تواصل ومراجعات وطلبات توثيق لهذه الجهات.

### العقارات والوسائط

يمثل جدول `estates` العنصر المركزي في المنصة، لأنه يحتوي على بيانات العقار الأساسية مثل النوع، السعر، الموقع، الحالة، والوصف. وترتبط به جداول الوسائط مثل `estate_images` و`estate_videos`، إضافة إلى جداول الإعلانات أو البيانات المرتبطة بالعقار.

### التفاعل والتواصل

تضم هذه المجموعة جداول مثل `favorite_estates`, `favorite_agents`, `messages`, و`notifications`. تساعد هذه الجداول على حفظ تفاعل المستخدم مع النظام، مثل حفظ عقار في المفضلة، إرسال رسالة، أو استلام إشعار.

### الاستثمار والتنبؤ السعري

تضم هذه المجموعة جداول مثل `investment_analyses`, `investment_portfolios`, `portfolio_properties`, و`price_predictions`. تمثل هذه الجداول الجانب الاستثماري والتحليلي في النظام، حيث يستطيع المستخدم تحليل عقار أو إضافته إلى محفظة أو طلب توقع سعري.

### التوصيات والتفضيلات

تشمل جداول مثل `user_preferences`, `property_interactions`, و`recommendations`. تحفظ هذه الجداول ما يفضله المستخدم وكيف يتفاعل مع العقارات، ثم تستخدم هذه البيانات لإنتاج اقتراحات أقرب لاهتمامه.

### الثقة والتقييمات والتوثيق

تضم هذه المجموعة جداول مثل `property_reviews`, `agent_reviews`, `company_reviews`, و`verification_requests`. تساعد هذه الجداول في بناء درجة ثقة داخل المنصة، وتسمح للمستخدمين بتقييم الجهات والعقارات، كما تسمح للإدارة بمراجعة طلبات التوثيق.

يمكن تلخيص أهم العلاقات في الجدول التالي:

| العلاقة | التوضيح |
|---|---|
| المدينة تمتلك عدة مناطق | كل مدينة يمكن أن تحتوي على أكثر من منطقة. |
| المنطقة تمتلك عدة عقارات | كل عقار يرتبط بموقع جغرافي محدد. |
| المستخدم يمتلك مفضلة | يمكن للمستخدم حفظ عقارات أو وسطاء. |
| العقار يمتلك وسائط | يمكن أن يحتوي العقار على صور وفيديوهات. |
| المستخدم يملك محفظة | يستطيع المستخدم إنشاء محافظ استثمارية. |
| المحفظة تضم عقارات | تربط جداول المحافظ بين المستخدم والعقارات المختارة. |
| المستخدم يرسل رسائل | الرسائل تربط المستخدمين أو الجهات العقارية ببعضها. |
| الجهة يمكن تقييمها | يمكن تقييم عقار أو وسيط أو شركة. |
| الجهة يمكن توثيقها | يراجع المدير طلبات التوثيق ويغير حالتها. |

## 4.4. البنية المعمارية للنظام

يعتمد النظام على بنية Client-Server مع خدمة تعلم آلي مستقلة. وهذا يعني أن المتصفح لا يتعامل مباشرة مع قاعدة البيانات، بل يتعامل مع واجهات API، وأن الخدمات المتخصصة يمكن فصلها عن النظام الرئيسي.

| الطبقة | التقنية | الدور |
|---|---|---|
| الواجهة الأمامية | Vue + Vite + Pinia + Vue Router | عرض الصفحات، إدارة الحالة، التنقل، والتفاعل مع المستخدم. |
| الواجهة الخلفية | Laravel + Sanctum | المصادقة، API، منطق الأعمال، الخدمات، والاختبارات. |
| قاعدة البيانات | MySQL أو SQLite حسب البيئة | تخزين المستخدمين والعقارات والعلاقات والبيانات التشغيلية. |
| خدمة التعلم الآلي | Flask + scikit-learn | استقبال بيانات العقار وإرجاع توقع السعر. |
| الخرائط | Leaflet + OpenStreetMap | عرض مواقع العقارات والبحث الجغرافي. |

### تدفق البيانات العام

عند تنفيذ عملية عادية، يضغط المستخدم على زر أو يفتح صفحة في الواجهة. ترسل Vue طلباً إلى Laravel عبر API. يتحقق Laravel من البيانات والصلاحيات، ثم يتعامل مع قاعدة البيانات أو مع خدمة داخلية، وبعدها يعيد استجابة JSON إلى الواجهة.

أما في حالة التنبؤ السعري، فإن الطلب يمر عبر Laravel أولاً، ثم يرسل Laravel البيانات إلى خدمة Flask. تقوم الخدمة بتحضير البيانات وتشغيل النموذج ثم تعيد السعر المتوقع، وبعدها يعرض Laravel النتيجة للمستخدم.

### تصميم المصادقة والصلاحيات

تعتمد المسارات الخاصة على Laravel Sanctum، بحيث لا يستطيع المستخدم تنفيذ عمليات مثل المفضلة، الرسائل، المحافظ، التحليلات، أو التقييمات إلا بعد تسجيل الدخول. كما توجد مسارات إدارية منفصلة يستخدمها المدير لمتابعة المحتوى وإدارته.

### تصميم الخدمات الخلفية

تم توزيع منطق الأعمال على خدمات خلفية مثل خدمات رفع الملفات، البحث الجغرافي، التوصيات، الاستثمار، المحافظ، الثقة، التنبؤ السعري، واتجاهات السوق. هذا الأسلوب يقلل ازدحام المتحكمات ويجعل كل خدمة مسؤولة عن مجال واضح.

## 4.5. تصميم الواجهات العامة ولوحات التحكم

تم تصميم الواجهة الأمامية على شكل صفحات ومكونات قابلة لإعادة الاستخدام. توجد صفحات عامة يستطيع الزائر الوصول إليها، وصفحات محمية تحتاج إلى تسجيل دخول، وصفحات إدارية خاصة بالمدير.

### الواجهات العامة

تشمل الواجهة العامة الصفحة الرئيسية، صفحة العقارات، تفاصيل العقار، المدن والمناطق، الشركات، الوسطاء، وصفحات العرض المرتبطة بالخريطة. هدف هذه الصفحات هو تمكين الزائر من الوصول السريع إلى العقارات والمعلومات الأساسية.

### واجهات المصادقة والملف الشخصي

تشمل صفحات تسجيل الدخول، التسجيل، وإدارة الملف الشخصي. بعد تسجيل الدخول يستطيع المستخدم تعديل معلوماته، إدارة روابط التواصل، متابعة مفضلته، واستخدام الوظائف المحمية.

### واجهات العقارات والمفضلة والتوصيات

تعرض هذه الواجهات بطاقات العقارات، التفاصيل، الصور، الفيديوهات، الموقع، وزر المفضلة. كما تعرض صفحة التوصيات عقارات مناسبة بناءً على تفضيلات المستخدم وتفاعلاته السابقة.

### واجهات الاستثمار والمحافظ

تسمح هذه الواجهات للمستخدم بإنشاء تحليل استثماري، عرض النتائج، إنشاء محفظة، وإضافة عقارات إلى المحفظة. وهي موجهة بشكل خاص للمستخدم الذي يتعامل مع العقار كفرصة استثمارية.

### لوحة مدير النظام

تحتوي لوحة الإدارة على صفحات لإدارة المستخدمين، العقارات، الشركات، الوسطاء، المدن، المناطق، التقييمات، طلبات التوثيق، والإحصاءات. تساعد هذه اللوحة على مراقبة النظام والحفاظ على جودة البيانات والمحتوى.

### المكونات المشتركة

تستخدم الواجهة مكونات مشتركة مثل البطاقات، الجداول، النماذج، أزرار الإجراءات، القوائم، التنبيهات، وعناصر التحميل. يساعد ذلك على توحيد شكل النظام وتقليل تكرار الكود.

## 4.6. خلاصة الدراسة التصميمية

أظهرت الدراسة التصميمية أن المشروع مبني على فصل واضح بين الواجهة الأمامية، الواجهة الخلفية، قاعدة البيانات، وخدمة التعلم الآلي. كما يوضح تصميم قاعدة البيانات أن النظام يغطي عدة مجالات مترابطة، مثل العقارات، المستخدمين، الشركات، الوسطاء، الوسائط، التوصيات، الاستثمار، الرسائل، الإشعارات، الثقة، والتوثيق.

تساعد مخططات التسلسل على فهم طريقة تنفيذ العمليات الرئيسية، بينما يوضح ERD العلاقات بين الكيانات، وتبين البنية المعمارية كيف تنتقل البيانات بين الطبقات. أما تصميم الواجهات فيقسم النظام إلى واجهات عامة، واجهات مستخدم، واجهات استثمار، ولوحة إدارة.

وبذلك يوفر التصميم أساساً منظماً لتنفيذ منصة عقارية قابلة للتوسع، ويمكن تطويرها مستقبلاً بإضافة وظائف مثل صلاحيات أدق، تحسينات أكبر للتوصيات، وتوسيع خدمة التنبؤ السعري.
