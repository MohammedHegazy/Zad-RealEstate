# الفصــل الثالث

# الدراسة التحليلية

## 3.1. توصيف المشروع

زاد للعقارات منصة ويب عربية تهدف إلى تنظيم عملية البحث عن العقارات والتفاعل معها ضمن نظام واحد. لا يقتصر النظام على عرض العقارات فقط، بل يوفر للمستخدم إمكانية البحث، مشاهدة التفاصيل، استخدام الخريطة، حفظ العقارات في المفضلة، التواصل، الحصول على توصيات، إجراء تحليلات استثمارية، ومتابعة محفظة عقارية.

يتكون النظام من ثلاث طبقات رئيسية. الطبقة الأولى هي الواجهة الأمامية المبنية باستخدام Vue وVite، وهي مسؤولة عن عرض الصفحات والتفاعل مع المستخدم. الطبقة الثانية هي الواجهة الخلفية المبنية باستخدام Laravel، وهي مسؤولة عن المصادقة، إدارة البيانات، واجهات API، الخدمات، والاختبارات. أما الطبقة الثالثة فهي خدمة مستقلة مبنية باستخدام Flask وscikit-learn للتنبؤ بسعر العقار اعتماداً على بياناته.

يعتمد النظام على عدة أنواع من المستخدمين: الزائر، المستخدم المسجل، المشتري أو المستثمر، مالك العقار، الوسيط العقاري، الشركة العقارية، ومدير النظام. لكل نوع من هؤلاء المستخدمين صلاحيات ووظائف مختلفة حسب دوره داخل المنصة.

## 3.2. مهام النظام الرئيسية

يمكن تلخيص مهام النظام الرئيسية بما يلي:

- تسجيل المستخدمين وتسجيل الدخول والخروج بطريقة آمنة.
- إدارة الملف الشخصي وروابط التواصل الخاصة بالمستخدم.
- عرض المدن والمناطق وربطها بالعقارات.
- تصفح العقارات ومشاهدة تفاصيلها وصورها وفيديوهاتها.
- البحث عن العقارات باستخدام الفلاتر والخريطة.
- حفظ العقارات أو الوسطاء في المفضلة.
- ضبط تفضيلات المستخدم لتوليد توصيات مناسبة.
- تسجيل تفاعلات المستخدم مع العقارات لتحسين الاقتراحات.
- إدارة عقارات المالك ووسائطها الأساسية.
- إدارة ملفات الشركات العقارية والوسطاء المرتبطين بها.
- إنشاء تحليلات استثمارية تساعد على تقييم العقار كفرصة مالية.
- إنشاء محافظ عقارية ومتابعة العقارات المرتبطة بها.
- طلب توقع سعر عقار من خدمة التنبؤ السعري.
- إرسال الرسائل واستقبال الإشعارات.
- إضافة تقييمات ومراجعات للعقارات والوسطاء والشركات.
- تقديم ومراجعة طلبات التوثيق ودرجات الثقة.
- إدارة النظام من لوحة المدير، بما في ذلك المستخدمون والعقارات والشركات والوسطاء والمراجعات.

هذه المهام تجعل النظام قريباً من منصة عقارية متكاملة، لأنه يجمع بين العرض، الإدارة، التواصل، التوصية، الثقة، والتحليل.

## 3.3. مستند المتطلبات

### 3.3.1. المتطلبات الوظيفية

المتطلبات الوظيفية هي الوظائف التي يجب أن يقدمها النظام للمستخدمين. ومن أهمها:

| الرقم | المتطلب |
|---|---|
| 1 | يجب أن يستطيع الزائر تصفح العقارات والمدن والمناطق والشركات والوسطاء. |
| 2 | يجب أن يستطيع المستخدم إنشاء حساب وتسجيل الدخول والخروج. |
| 3 | يجب أن يستطيع المستخدم المسجل تعديل ملفه الشخصي وروابط التواصل. |
| 4 | يجب أن يستطيع المستخدم مشاهدة تفاصيل العقار ووسائطه وموقعه على الخريطة. |
| 5 | يجب أن يستطيع المستخدم حفظ العقارات أو الوسطاء في المفضلة. |
| 6 | يجب أن يستطيع المستخدم ضبط تفضيلاته للحصول على توصيات مناسبة. |
| 7 | يجب أن يستطيع النظام إنشاء توصيات اعتماداً على التفضيلات والتفاعلات. |
| 8 | يجب أن يستطيع المستثمر إنشاء تحليل استثماري لعقار معين. |
| 9 | يجب أن يستطيع المستخدم إنشاء محفظة عقارية وإضافة عقارات إليها. |
| 10 | يجب أن يستطيع المستخدم إرسال رسائل ومتابعة الإشعارات. |
| 11 | يجب أن يستطيع المستخدم طلب توقع سعر عقار. |
| 12 | يجب أن يدعم النظام تقييم العقارات والوسطاء والشركات. |
| 13 | يجب أن يدعم النظام طلبات التوثيق ودرجات الثقة. |
| 14 | يجب أن يستطيع المدير إدارة المستخدمين والمحتوى والطلبات من لوحة الإدارة. |

### 3.3.2. المتطلبات غير الوظيفية

أما المتطلبات غير الوظيفية فهي الشروط العامة التي تؤثر في جودة النظام وتشغيله، ومن أهمها:

- فصل الواجهة الأمامية عن الواجهة الخلفية من خلال REST API.
- استخدام JSON في تبادل البيانات بين Vue وLaravel.
- حماية المسارات الخاصة باستخدام Laravel Sanctum.
- تنظيم الكود إلى Controllers وServices وModels لتسهيل الصيانة.
- دعم واجهة عربية مناسبة للاستخدام باتجاه RTL.
- التعامل مع الصور والفيديوهات بطريقة منظمة.
- استخدام الخريطة لعرض المواقع العقارية بوضوح.
- فصل خدمة التنبؤ السعري عن النظام الرئيسي لتسهيل تطويرها.
- توفير اختبارات تغطي أجزاء مهمة من منطق النظام.
- تصميم قاعدة بيانات قابلة للتوسع وتضم علاقات واضحة بين الكيانات.

## 3.4. مخططات حالات الاستخدام

تعتمد حالات الاستخدام في النظام على مجموعة من الممثلين الرئيسيين. يمثل كل ممثل نوعاً من المستخدمين أو الأطراف التي تتعامل مع النظام.

| الممثل | أهم حالات الاستخدام |
|---|---|
| الزائر | تصفح العقارات، مشاهدة المدن والمناطق، عرض الشركات والوسطاء، فتح تفاصيل العقار. |
| المستخدم المسجل | إدارة الحساب، حفظ المفضلة، إرسال الرسائل، متابعة الإشعارات، كتابة تقييم. |
| المشتري أو المستثمر | البحث المتقدم، استخدام التوصيات، إنشاء تحليل استثماري، إدارة محفظة عقارية. |
| مالك العقار | إنشاء عقار، تعديل بيانات العقار، إدارة الصور والفيديوهات، متابعة حالة العقار. |
| الوسيط العقاري | عرض ملف الوسيط، إدارة معلوماته، استقبال الرسائل، الارتباط بالشركات أو العقارات. |
| الشركة العقارية | إدارة ملف الشركة، إدارة الوسطاء، متابعة العقارات، طلب التوثيق. |
| مدير النظام | إدارة المستخدمين، العقارات، الشركات، الوسطاء، التقييمات، التوثيق، والإحصاءات. |

يوضح هذا التقسيم أن النظام لا يخدم مستخدماً واحداً فقط، بل يدعم عدة أدوار مترابطة داخل المجال العقاري. فالزائر يبدأ بالتصفح، والمستخدم المسجل يتفاعل، والمستثمر يحلل، والمالك أو الشركة يضيفان المحتوى، والمدير يراقب وينظم.

## 3.5. حالات الاستخدام المختصرة

فيما يلي أهم حالات الاستخدام بصورة مختصرة:

### تصفح العقارات

يدخل المستخدم إلى صفحة العقارات، فيعرض النظام قائمة العقارات المتاحة مع إمكانية التصفية حسب المدينة أو النوع أو السعر أو الموقع. يستطيع المستخدم فتح صفحة التفاصيل لمشاهدة معلومات العقار وصوره وموقعه.

### البحث على الخريطة

يفتح المستخدم الخريطة ويحدد منطقة أو نطاقاً معيناً، فيرسل النظام الطلب إلى الواجهة الخلفية، ثم يعيد العقارات القريبة أو المطابقة للشروط. تساعد هذه الحالة المستخدم على فهم موقع العقار بصرياً.

### تسجيل الدخول

يدخل المستخدم بياناته، ثم يتحقق النظام منها عبر Laravel Sanctum. عند نجاح العملية يستطيع المستخدم الوصول إلى الوظائف المحمية مثل المفضلة والرسائل والتوصيات والمحافظ.

### إضافة عقار إلى المفضلة

عندما يجد المستخدم عقاراً مهماً، يضغط على زر المفضلة. يتحقق النظام من تسجيل الدخول ثم يحفظ العلاقة بين المستخدم والعقار، وبعدها يظهر العقار ضمن صفحة المفضلة.

### ضبط التفضيلات وتوليد التوصيات

يضبط المستخدم اهتماماته مثل المدينة، نوع العقار، السعر، أو الغرض من الاستخدام. يستخدم النظام هذه البيانات مع تفاعلات المستخدم السابقة لتوليد قائمة عقارات مقترحة.

### إنشاء عقار من قبل المالك

يدخل المالك بيانات العقار الأساسية ويرفع الصور أو الفيديوهات المناسبة. يحفظ النظام العقار ويربطه بالمالك أو الجهة المسؤولة، ثم يمكن للمدير متابعة حالته أو مراجعته حسب سياسة النظام.

### إدارة ملف الشركة والوسطاء

تستطيع الشركة العقارية إدارة بياناتها وروابط التواصل الخاصة بها، كما يمكن ربط الوسطاء بها. يساعد ذلك على عرض الجهة العقارية بشكل منظم وموثوق للمستخدمين.

### إنشاء تحليل استثماري

يختار المستخدم عقاراً ويدخل المعطيات المطلوبة للتحليل. يقوم النظام بحساب مؤشرات مثل العائد المتوقع أو جدوى الاستثمار، ثم يعرض النتيجة للمستخدم بطريقة تساعده على المقارنة.

### إضافة عقار إلى محفظة

ينشئ المستثمر محفظة عقارية ثم يضيف إليها عقارات محددة. يسمح ذلك بمتابعة مجموعة عقارات معاً بدلاً من التعامل مع كل عقار بشكل منفصل.

### إرسال رسالة

يرسل المستخدم رسالة إلى جهة مرتبطة بالعقار أو إلى طرف آخر داخل النظام. يحفظ النظام الرسالة ويرسل إشعاراً للطرف المستلم ليتمكن من متابعتها.

### طلب توقع سعر عقار

يرسل المستخدم بيانات العقار إلى النظام، ثم يحول Laravel الطلب إلى خدمة Flask الخاصة بالتنبؤ السعري. تعيد الخدمة السعر المتوقع ليتم عرضه للمستخدم.

### كتابة تقييم ومراجعته

يستطيع المستخدم تقييم عقار أو وسيط أو شركة. تخزن المراجعة في النظام، ويمكن للإدارة مراجعتها عند الحاجة للحفاظ على جودة المحتوى وثقة المستخدمين.

### مراجعة طلب توثيق

تقدم الشركة أو الجهة العقارية طلب توثيق. يراجع المدير الطلب والبيانات المرتبطة به، ثم يوافق عليه أو يرفضه، مما يؤثر على درجة الثقة أو حالة الجهة.

### إدارة المحتوى من لوحة المدير

يدخل المدير إلى لوحة الإدارة لمتابعة المستخدمين والعقارات والشركات والوسطاء والتقييمات وطلبات التوثيق. يستطيع المدير تعديل الحالات ومراجعة المحتوى وحذف العناصر غير المناسبة.

## 3.6. مخططات النشاط

توضح مخططات النشاط طريقة انتقال العمليات داخل النظام من البداية حتى النهاية. وقد تم إعداد مخططات في مجلد `assets` لعدد من المسارات المهمة، ومن أهمها:

- نشاط جلب التوصيات: يبدأ من طلب المستخدم ثم قراءة التفضيلات والتفاعلات وإرجاع العقارات المناسبة.
- نشاط تقييم عقار داخل محرك التوصيات: يحسب النظام مدى ملاءمة العقار لتفضيلات المستخدم.
- نشاط إنشاء عقار: يبدأ بإدخال البيانات، ثم التحقق منها، ثم حفظ العقار ووسائطه.
- نشاط إنشاء تحليل استثماري: يستقبل النظام بيانات التحليل، ينفذ الحسابات، ثم يعرض النتيجة.
- نشاط إضافة عقار إلى محفظة: يختار المستخدم المحفظة والعقار ثم يحفظ النظام العلاقة بينهما.
- نشاط لوحة المستثمر: يعرض النظام ملخص المحافظ والتحليلات والبيانات الاستثمارية.
- نشاط إرسال رسالة: يحفظ النظام الرسالة ويرسل إشعاراً للطرف المستلم.
- نشاط التنبؤ بسعر عقار: ينتقل الطلب من الواجهة إلى Laravel ثم إلى خدمة Flask ثم تعود النتيجة.
- نشاط اتجاهات السوق: يجمع النظام بيانات الأسعار أو المناطق ويعرض مؤشرات تساعد في فهم السوق.
- نشاط اعتماد العقارات والشركات: يراجع المدير الطلبات ثم يغير الحالة إلى مقبول أو مرفوض.
- نشاط الإشعارات: ينشئ النظام إشعاراً عند حدوث عملية مهمة، ثم يعرضه للمستخدم.
- نشاط مراجعة التقييمات والتوثيق: يتابع المدير المحتوى المرتبط بالثقة ويتخذ القرار المناسب.

تساعد هذه المخططات على فهم النظام عملياً، لأنها لا تعرض الكيانات فقط، بل توضح تسلسل العمل بين المستخدم والواجهة الأمامية وواجهة API والخدمات الداخلية. وبذلك تكون الدراسة التحليلية قد حددت أدوار النظام، متطلباته، حالاته الأساسية، وسير عملياته المهمة.
