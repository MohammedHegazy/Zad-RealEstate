# الفصــل الثامن

# الأدوات واللغات المستخدمة

## 8.1. مقدمة

يعتمد مشروع **زاد للعقارات** على مجموعة من الأدوات واللغات التي تخدم طبيعة النظام كمنصة ويب متعددة الطبقات. فالواجهة الأمامية تحتاج إلى إطار تفاعلي لبناء صفحات البحث والعقارات ولوحة الإدارة، والخادم الخلفي يحتاج إلى إطار قوي لبناء REST API والتحقق من المدخلات والصلاحيات، وخدمة التنبؤ السعري تحتاج إلى بيئة Python مناسبة لتشغيل نموذج تعلم آلي.

لذلك تم استخدام Vue وVite في الواجهة، Laravel وSanctum في الخلفية، MySQL أو SQLite في قاعدة البيانات، وFlask مع scikit-learn في خدمة الذكاء الاصطناعي. كما تم استخدام أدوات مساعدة مثل Composer وnpm وArtisan وPHPUnit وDBML وMarkdown لإدارة الحزم والتشغيل والاختبار والتوثيق.

## 8.2. النموذج المستخدم في تطوير المشروع

### 8.2.1. أسلوب التطوير التكراري

تم التعامل مع المشروع بأسلوب تطوير تكراري، حيث يتم تنفيذ الوظائف الأساسية أولاً ثم إضافة الوظائف الأكثر تقدماً تدريجياً. بدأ العمل من بنية قاعدة البيانات والمصادقة والكيانات الأساسية مثل المستخدمين والعقارات والمدن والمناطق، ثم انتقل إلى الشركات والوسطاء، وبعد ذلك أضيفت وظائف مثل المفضلة، التوصيات، الاستثمار، الرسائل، الثقة، والتنبؤ السعري.

هذا الأسلوب مناسب للمشروع لأن النظام يحتوي على وحدات متعددة، ومن الصعب تنفيذها كلها دفعة واحدة. كما أن كل وحدة تعتمد على وحدات سابقة، فالتوصيات تحتاج إلى عقارات وتفضيلات ومفضلة، والاستثمار يحتاج إلى عقارات وأسعار وبيانات إيجار، والثقة تحتاج إلى مستخدمين ووسطاء وشركات وتقييمات.

### 8.2.2. سبب اختيار هذا الأسلوب

تم اختيار التطوير التكراري للأسباب التالية:

- إمكانية اختبار كل وحدة بعد تنفيذها.
- تقليل التعقيد الناتج عن تعدد الأدوار.
- القدرة على تعديل قاعدة البيانات أو الواجهة عند ظهور حاجة جديدة.
- سهولة بناء التوثيق والمخططات بالتوازي مع التنفيذ.
- إمكانية إضافة وظائف متقدمة مثل ML بعد اكتمال الأساس العقاري.

## 8.3. النموذج المعماري للنظام

### 8.3.1. نموذج Client/API

يعتمد النظام على نموذج Client/API. الواجهة الأمامية هي تطبيق Vue مستقل يعمل في المتصفح، ولا يتعامل مباشرة مع قاعدة البيانات. كل عملية جلب أو حفظ أو تعديل تتم عبر طلب HTTP إلى Laravel API. يعيد الخادم استجابات JSON، ثم تعرضها الواجهة داخل الصفحات والمكونات.

هذا النموذج مناسب لأن المشروع يحتوي على واجهة أمامية واضحة ولوحة إدارة، ويمكن مستقبلاً إضافة تطبيق موبايل يستخدم نفس API دون إعادة بناء الخادم.

### 8.3.2. نموذج MVC في Laravel

يستخدم Laravel نموذج MVC بشكل واضح:

- **Model:** مثل `User`, `Estate`, `Companies`, `Agent`, `Recommendation`, و`InvestmentPortfolio`.
- **Controller:** مثل `EstateController`, `CompanyController`, `RecommendationController`, و`TrustModerationController`.
- **View:** لا يستخدم Laravel هنا كواجهة Blade أساسية، لأن الواجهة الرئيسية مبنية بـ Vue، لكن Laravel يعيد JSON بدلاً من صفحات HTML.

كما يستخدم المشروع Services لعزل منطق الأعمال المعقد عن المتحكمات، مثل `InvestmentCalculatorService`, `RecommendationService`, و`TrustScoreService`.

### 8.3.3. REST API

تم تصميم الخادم الخلفي كـ REST API. توجد مسارات عامة ومحمية وإدارية:

- `/api/v1`: للمستخدمين والوظائف العامة.
- `/api/v1/admin`: للمدير.

وتستخدم المسارات أساليب HTTP مثل GET للعرض، POST للإنشاء، PUT/PATCH للتعديل، وDELETE للحذف. هذا يجعل سلوك API واضحاً وقابلاً للتوثيق والاختبار.

### 8.3.4. خدمة ML منفصلة

خدمة التنبؤ السعري ليست جزءاً مباشراً من Laravel، بل تعمل كخدمة Flask مستقلة. يتواصل Laravel معها عبر HTTP عند طلب التنبؤ. هذا النموذج يفصل مسؤوليات PHP عن مسؤوليات Python، ويسمح بتطوير نموذج التسعير أو استبداله دون التأثير على بقية الخادم.

## 8.4. بيئة التطوير

تم استخدام بيئة تطوير محلية يمكنها تشغيل ثلاث خدمات:

| الجزء | البيئة |
|---|---|
| الخادم الخلفي | PHP 8.2، Composer، Laravel 12 |
| الواجهة الأمامية | Node.js، npm، Vite، Vue |
| قاعدة البيانات | MySQL أو SQLite |
| خدمة ML | Python، Flask، scikit-learn |
| الاختبارات | PHPUnit |
| التوثيق | Markdown، Mermaid، DBML |

يحتاج المطور إلى تشغيل Laravel API، ثم تشغيل Vite للواجهة، وتشغيل Flask عند استخدام التنبؤ السعري.

## 8.5. الأدوات المستخدمة

### 8.5.1. Visual Studio Code

يعد Visual Studio Code مناسباً لتحرير ملفات PHP وJavaScript وVue وPython وMarkdown. يساعد في التنقل بين ملفات المشروع، البحث داخل الكود، وإدارة الطرفية المدمجة. وبسبب أن المشروع يحتوي على أكثر من لغة، فإن محرر يدعم الإضافات المتعددة يكون مفيداً جداً.

### 8.5.2. Composer

يستخدم Composer لإدارة حزم PHP الخاصة بالخادم الخلفي. من خلاله يتم تثبيت Laravel وSanctum وPHPUnit وباقي حزم التطوير. أهم الأوامر:

```bash
composer install
composer test
```

كما يحتوي `composer.json` على سكربتات مثل `setup`, `dev`, و`test` لتسهيل التشغيل.

### 8.5.3. Node.js وnpm

يستخدم Node.js وnpm لإدارة الواجهة الأمامية. يتم تثبيت حزم Vue وVite وPinia وBootstrap وLeaflet من خلال npm. أهم الأوامر:

```bash
npm install
npm run dev
npm run build
npm run lint
```

### 8.5.4. Vite

Vite هو أداة تشغيل وبناء الواجهة الأمامية. يوفر خادم تطوير سريع للواجهة، ويستخدم في بناء الملفات النهائية للإنتاج. يعتمد المشروع عليه داخل `project-RealEstate`.

### 8.5.5. Laravel Artisan

Artisan هو أداة الأوامر الخاصة بـ Laravel. يستخدم لإنشاء مفتاح التطبيق، تشغيل الخادم، تنفيذ migrations، تشغيل الاختبارات، ومسح الإعدادات. أمثلة:

```bash
php artisan key:generate
php artisan migrate --seed
php artisan serve
php artisan test
php artisan config:clear
```

### 8.5.6. Git

يستخدم Git لإدارة إصدارات المشروع ومتابعة التعديلات. وهو مهم خصوصاً في مشروع يحتوي على واجهة وخلفية وخدمة ML وتوثيق، لأن التعديلات كثيرة وموزعة على مجلدات متعددة.

### 8.5.7. Postman أو أدوات اختبار API

يمكن استخدام Postman أو أي أداة مشابهة لاختبار API يدوياً، مثل تسجيل الدخول، إرسال توكن، إنشاء عقار، تجربة التوصيات، أو اختبار مسارات الإدارة. ورغم وجود اختبارات PHPUnit، تبقى أدوات API مفيدة أثناء التطوير السريع وفحص الاستجابات.

أمثلة على الطلبات التي يمكن اختبارها يدوياً:

- `POST /api/v1/login`
- `GET /api/v1/estates`
- `GET /api/v1/estates/map`
- `GET /api/v1/recommendations`
- `POST /api/v1/price-predictions/preview`
- `GET /api/v1/admin/statistics`

### 8.5.8. DBML وdbdiagram

يستخدم DBML لتوثيق مخطط قاعدة البيانات. يحتوي المشروع على ملف `ERD2.dbml` الموصوف بأنه مطابق للمهاجرات. يمكن استيراد هذا الملف إلى dbdiagram لعرض الجداول والعلاقات بصرياً.

### 8.5.9. Mermaid

يستخدم Mermaid داخل ملفات Markdown لرسم مخططات مثل مخططات النشاط، التسلسل، الخرائط الذهنية، ومخطط Gantt. هذا مناسب للتوثيق لأنه يبقي المخطط نصياً وقابلاً للتعديل داخل المستودع.

### 8.5.10. PHPUnit

يستخدم PHPUnit لاختبار الخادم الخلفي. يحتوي المشروع على اختبارات Feature وUnit تغطي الإدارة، التوصيات، الاستثمار، الثقة، الروابط الاجتماعية، الإشعارات، والبحث الجغرافي.

### 8.5.11. Laravel Pint

Laravel Pint موجود ضمن حزم التطوير في `composer.json`. يستخدم لتنسيق كود PHP وفق أسلوب Laravel. يساعد هذا على جعل ملفات PHP أكثر اتساقاً عند العمل على المشروع.

### 8.5.12. Python pip

يستخدم pip لتثبيت متطلبات خدمة التنبؤ السعري من ملف `requirements.txt`. أهم الأمر:

```bash
pip install -r requirements.txt
```

## 8.6. اللغات المستخدمة في المشروع

### 8.6.1. PHP

تستخدم PHP في الخادم الخلفي من خلال Laravel. بها تم تنفيذ المتحكمات، النماذج، الطلبات، الخدمات، الاختبارات، وإعدادات النظام. وتعد PHP اللغة الأساسية لمنطق الأعمال وقاعدة البيانات.

### 8.6.2. JavaScript

تستخدم JavaScript في الواجهة الأمامية من خلال Vue وملفات API وcomposables. بها يتم التعامل مع الحالة، استدعاء API، تنسيق البيانات، وإدارة تفاعل المستخدم داخل المتصفح.

### 8.6.3. Vue Single File Components

تستخدم ملفات `.vue` لبناء الصفحات والمكونات. يحتوي كل ملف غالباً على template وscript وأحياناً style. مثال ذلك صفحات العقارات ولوحة الإدارة والمكونات مثل النماذج والجداول.

### 8.6.4. Python

تستخدم Python في خدمة التنبؤ السعري. يقوم `server.py` بتشغيل Flask، استقبال JSON، تجهيز الخصائص، استخدام النموذج المحفوظ، وإرجاع السعر المتوقع.

### 8.6.5. HTML

يظهر HTML داخل قوالب Vue، ويستخدم لبناء هيكل الصفحات والمكونات. وبما أن التطبيق عربي، فإن بنية الواجهة تراعي اتجاه RTL في التصميم العام.

### 8.6.6. CSS

تستخدم CSS لتنسيق الواجهة العامة وصفحات الإدارة والمكونات. يحتوي المشروع على ملفات مثل `styles/index.css`, `styles/components`, و`styles/pages`. كما يستخدم Bootstrap كأساس لبعض عناصر الواجهة.

### 8.6.7. SQL

يستخدم SQL بشكل غير مباشر عبر Laravel migrations وEloquent، وبشكل مباشر في بعض الاستعلامات التحليلية مثل اتجاهات السوق. قاعدة البيانات تحفظ العلاقات الأساسية بين المستخدمين والعقارات والشركات والوسطاء والتوصيات والمحافظ.

### 8.6.8. Markdown

يستخدم Markdown لتوثيق المشروع والفصول والمخططات. كل فصول التقرير مكتوبة بصيغة Markdown، مما يجعلها سهلة القراءة والتعديل والتحويل لاحقاً إلى صيغ أخرى.

### 8.6.9. DBML

يستخدم DBML لتوثيق ERD. وهو مناسب لأنه يسمح بوصف الجداول والعلاقات والفهارس بشكل نصي واضح.

## 8.7. تقنيات ومكتبات الخادم الخلفي Backend

### 8.7.1. Laravel 12

Laravel هو الإطار الأساسي للخادم الخلفي. يوفر Routing، Controllers، Middleware، Validation، Eloquent ORM، Migrations، Factories، Testing، وملفات إعدادات منظمة. في المشروع، تم استخدام Laravel لبناء API وليس لتقديم واجهات Blade رئيسية.

### 8.7.2. Laravel Sanctum

يستخدم Sanctum للمصادقة عبر التوكن. عند تسجيل الدخول أو التسجيل، ينشئ النظام token للمستخدم، ثم تستخدمه الواجهة في الطلبات المحمية. وتحمي المسارات الخاصة بوسيط `auth:sanctum`.

### 8.7.3. Middleware

يستخدم Laravel Middleware لحماية المسارات. أهم مثال هو وسيط `admin` الذي يمنع المستخدم غير المدير من الوصول إلى `/api/v1/admin`. كما يستخدم `auth:sanctum` لحماية مسارات المستخدم المسجل.

### 8.7.4. Eloquent ORM

يستخدم Eloquent للتعامل مع الجداول كنماذج. توجد نماذج مثل `Estate`, `Companies`, `Agent`, `Recommendation`, `InvestmentAnalysis`, و`VerificationRequest`. تساعد العلاقات في Eloquent على جلب البيانات المرتبطة مثل صور العقار أو تقييمات الوسيط.

### 8.7.5. Laravel Migrations وFactories

تستخدم migrations لتعريف بنية قاعدة البيانات، مثل جداول العقارات والشركات والوسطاء والتوصيات والمحافظ. وتستخدم factories في الاختبارات لإنشاء بيانات تجريبية بسرعة.

### 8.7.6. Form Requests

يستخدم المشروع Form Requests للتحقق من المدخلات. توجد طلبات مثل `StoreEstateRequest`, `UpdateEstateRequest`, `StoreInvestmentAnalysisRequest`, `StoreMessageRequest`, وطلبات الثقة والإدارة. هذا يمنع تكرار قواعد التحقق داخل المتحكمات.

### 8.7.7. Services

تم استخدام Services لعزل منطق الأعمال. من أهم الخدمات:

- `InvestmentCalculatorService`
- `RecommendationService`
- `RecommendationGeneratorService`
- `RecommendationScoringService`
- `GeoSearchService`
- `PortfolioService`
- `TrustScoreService`
- `PricePredictionClient`
- `EstatePricePredictionService`

وجود هذه الخدمات يجعل المتحكمات أخف، لأن المتحكم لا يحتاج إلى معرفة تفاصيل حساب ROI أو طريقة ترتيب التوصيات أو كيفية احتساب درجة الثقة. كما يجعل الاختبار أسهل، لأن بعض الخدمات يمكن اختبارها بوحدات Unit Tests دون إرسال طلب HTTP كامل.

### 8.7.8. Resources وTraits

يستخدم المشروع Resources وTraits لتنسيق بعض الاستجابات، خصوصاً في مجالات الثقة والمحافظ وتنسيق بيانات العقارات والشركات والوسطاء. هذا يساعد على إرجاع JSON بشكل منظم ومتكرر.

من أمثلة ذلك:

- موارد الثقة مثل `TrustScoreResource`, `PropertyReviewResource`, و`VerificationRequestResource`.
- موارد المحافظ مثل `PortfolioResource`, `PortfolioItemResource`, و`DashboardSummaryResource`.
- Traits مثل `FormatsEstateResponse`, `FormatsCompanyResponse`, و`FormatsRecommendationResponse`.

هذه الطبقة مهمة لأن الواجهة لا يجب أن تتعامل مع شكل بيانات متغير من متحكم لآخر.

### 8.7.9. Laravel Filesystem

يستخدم Laravel Filesystem لإدارة الملفات المرفوعة مثل صور العقارات، الفيديوهات، صور المدن، صور الشركات، ومستندات التوثيق. ويتم حفظ مسارات الملفات في قاعدة البيانات.

### 8.7.10. PHPUnit وLaravel Test Runner

يستخدم PHPUnit مع Laravel Test Runner لاختبار المسارات والخدمات. الاختبارات تغطي حالات صحيحة وحالات رفض مثل عدم امتلاك صلاحية أو محاولة تكرار بيانات.

أمثلة على مجالات الاختبار:

- الإدارة: المستخدمون، العقارات، الشركات، الوسطاء، المواقع.
- التوصيات: التفضيلات، العقارات المشابهة، حساب الدرجة.
- الاستثمار: التحليلات، المحافظ، لوحة المستثمر.
- الثقة: المراجعات، التوثيق، درجة الثقة.
- الروابط الاجتماعية: تعدد الأشكال ومنع التكرار.

## 8.8. تقنيات ومكتبات الواجهة الأمامية Frontend

### 8.8.1. Vue

Vue هو الإطار الأساسي للواجهة الأمامية. تم استخدامه لبناء صفحات العقارات، المدن، المناطق، الشركات، الوسطاء، الملف الشخصي، المفضلة، التوصيات، ولوحة الإدارة. تعتمد الصفحات على المكونات وملفات API لجلب البيانات.

### 8.8.2. Vite

Vite يستخدم لتشغيل الواجهة أثناء التطوير وبناء نسخة الإنتاج. يتم تشغيله عبر `npm run dev` والبناء عبر `npm run build`.

### 8.8.3. Vue Router

يدير Vue Router مسارات الواجهة مثل `/estates`, `/estates/map`, `/companies`, `/profile`, و`/admin/dashboard`. كما يستخدم guards لحماية الصفحات المحمية أو الإدارية.

### 8.8.4. Pinia

تستخدم Pinia لإدارة حالة التطبيق، خصوصاً حالة المصادقة والمستخدم الحالي. تساعد في معرفة هل المستخدم مسجل الدخول وهل يمكنه الوصول إلى صفحات معينة.

### 8.8.5. Bootstrap

يستخدم Bootstrap كأساس لبعض عناصر الواجهة والتنسيق العام. كما يستخدم المشروع Bootstrap Icons لتوفير أيقونات جاهزة.

### 8.8.6. Leaflet

يستخدم Leaflet لعرض الخرائط والعقارات حسب الإحداثيات. يرتبط ذلك بواجهات API الخاصة بالخريطة والبحث الجغرافي.

### 8.8.7. ملفات API في الواجهة

يحتوي مجلد `src/api` على ملفات مثل `estates.js`, `companies.js`, `agents.js`, `recommendations.js`, و`pricePredictions.js`. كما يحتوي `src/api/admin` على ملفات خاصة بلوحة الإدارة. هذا الفصل يجعل استدعاءات الخادم منظمة بدلاً من كتابتها مباشرة داخل كل صفحة.

أمثلة على تقسيم ملفات API:

| الملف | المجال |
|---|---|
| `auth.js` | تسجيل الدخول والتسجيل |
| `estates.js` | العقارات والتفاصيل والخريطة |
| `favorites.js` | المفضلة |
| `recommendations.js` | التوصيات والعقارات المشابهة |
| `reviews.js` | التقييمات |
| `pricePredictions.js` | التنبؤ السعري |
| `admin/estates.js` | إدارة العقارات |
| `admin/trust.js` | مراجعة الثقة والتوثيق |

### 8.8.8. Composables

تستخدم composables لإعادة استخدام منطق الجلب والمعالجة، مثل `useEstatesList`, `useEstateDetail`, `useAgentsList`, و`useAdminDashboard`. هذا يقلل تكرار الكود بين الصفحات.

وجود composables مفيد في الصفحات التي تحتاج إلى حالة تحميل، أخطاء، صفحات متعددة، وطلبات متكررة. فبدلاً من كتابة نفس المنطق داخل كل صفحة، يتم عزله في ملف قابل لإعادة الاستخدام.

### 8.8.9. مكونات الواجهة المشتركة

يحتوي المشروع على مكونات مشتركة مثل `AppButton`, `AppInput`, `Pagination`, `ErrorAlert`, `LoadingSpinner`, `TrustBadge`, و`StarRating`. كما توجد مكونات إدارية مثل `AdminDataTable`, `AdminEstateForm`, و`AdminModerationDialog`.

ومن الناحية العملية، تسمح هذه المكونات بتوحيد تجربة المستخدم. فزر الحفظ، رسالة الخطأ، جدول الإدارة، شارة الحالة، وتقييم النجوم تظهر بشكل متناسق في أكثر من صفحة، مما يجعل الواجهة أسهل استخداماً وصيانة.

## 8.9. الخدمات الخارجية المستخدمة

### 8.9.1. OpenStreetMap وLeaflet

تستخدم الخرائط اعتماداً على Leaflet وOpenStreetMap. يوفر النظام إعدادات مزود الخريطة داخل `config/realestate.php`، وتستخدم الواجهة هذه البيانات لعرض الخريطة.

### 8.9.2. Flask Price Prediction Service

تعد خدمة Flask خدمة خارجية بالنسبة إلى Laravel، رغم أنها موجودة داخل نفس المستودع. تستقبل خصائص العقار وتعيد السعر المتوقع. يعتمد تشغيلها على Python وملفات النموذج المحفوظة.

أهم ملفات الخدمة:

| الملف | الدور |
|---|---|
| `server.py` | تشغيل Flask واستقبال `/predict` |
| `requirements.txt` | متطلبات Python |
| `real_estate_model.pkl` | النموذج المحفوظ |
| `label_encoder.pkl` | تحويل الموقع إلى قيمة رقمية |

### 8.9.3. scikit-learn Model

يستخدم نموذج scikit-learn المحفوظ لتقدير السعر. يتم تحميله عبر joblib داخل خدمة Flask. وتستخدم خدمة LabelEncoder لتحويل قيمة الموقع إلى قيمة رقمية مناسبة للنموذج.

### 8.9.4. خدمات قابلة للإضافة مستقبلاً

لا يحتوي المشروع حالياً على دفع إلكتروني أو SMS أو بريد إنتاجي متقدم كجزء أساسي من التنفيذ، لكن يمكن إضافة خدمات مستقبلية مثل:

- دفع إلكتروني للإعلانات المميزة.
- رسائل SMS للتواصل.
- بريد إلكتروني للتنبيهات.
- تخزين خارجي للوسائط.
- خدمة خرائط مدفوعة عند الحاجة.

## 8.10. قاعدة البيانات والتخزين

### 8.10.1. MySQL أو SQLite

يدعم Laravel استخدام MySQL أو SQLite حسب إعدادات `.env`. في التطوير يمكن استخدام SQLite لتسهيل الاختبارات، بينما يناسب MySQL بيئات التشغيل الأكبر. تعتمد بنية الجداول على migrations، وتم توثيقها في `ERD2.dbml`.

### 8.10.2. Laravel Migrations

توجد migrations لتعريف جداول المستخدمين، المدن، المناطق، العقارات، الشركات، الوسطاء، الوسائط، المفضلة، الرسائل، الإشعارات، الاستثمار، التوصيات، التقييمات، والتوثيق. تساعد migrations على إعادة بناء قاعدة البيانات بشكل منظم.

### 8.10.3. Laravel Factories وSeeders

تستخدم factories لإنشاء بيانات تجريبية في الاختبارات. كما توجد seeders مثل `AdminSeeder` لإنشاء بيانات إدارية أو بيانات أولية عند الحاجة.

### 8.10.4. تخزين الوسائط

يتم تخزين الصور والفيديوهات ومستندات التوثيق في نظام ملفات Laravel، بينما تحفظ المسارات داخل قاعدة البيانات. ويحتاج عرض الملفات العامة إلى إعداد التخزين المناسب مثل `storage:link` عند استخدام القرص العام.

أنواع الملفات المستخدمة في المشروع تشمل:

- صور العقارات.
- فيديوهات العقارات.
- إعلانات العقارات.
- صور المدن.
- صور الشركات.
- صور الوسطاء.
- مستندات التوثيق.

وتحدد إعدادات `config/realestate.php` حدوداً للرفع مثل حجم الصور والفيديوهات والامتدادات المسموحة.
